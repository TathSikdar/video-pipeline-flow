# Implementation Logs

This file contains a historical log of all implementation choices and technical reasoning for the Video Pipeline Flow project.

## Initialization
- **Action**: Initialized monorepo structure with `frontend/` and `backend/`.
- **Reasoning**: To maintain a unified codebase while explicitly decoupling the architecture.
- **Action**: Configured `/dev/shm` mapping via Docker `shm_size` for backend.
- **Reasoning**: To fulfill the strict RAM Disk `tmpfs` requirement for `yt-dlp` and `ffmpeg` multiplexing locally, bypassing "No space left on device" errors without triggering unprivileged blockages on Render.
- **Action**: Selected Python 3.11 and Tailwind v3.
- **Reasoning**: Python 3.11 provides maximum performance while avoiding standard library removal breaks in 3.12. Tailwind v3 ensures maximum plugin compatibility.

## Phase 2: Core Download Pipeline
- **Action**: Installed `aria2` via Dockerfile and configured `yt-dlp` to utilize it.
- **Reasoning**: Bypass YouTube's single HTTP connection throttle by establishing 16 concurrent TCP byte-range connections, maximizing download bandwidth.
- **Action**: Created Python garbage collection wrappers and configured Strict Synchronous Blocking.
- **Reasoning**: Prevents FFmpeg from concurrently multiplexing massive 4K/Audio files which would instantly crash the 4GB `/dev/shm` RAM disk. Synchronous limits guarantee exactly 1 video memory footprint at any given time.
- **Action**: Injected `SabrStreamingAdapter` and `SabrUmpProcessor` classes.
- **Reasoning**: Reverses the Universal Media Protocol (UMP) and server-side fake buffering algorithms to bypass the SABR 360p resolution downgrade enforced on headless clients.

## Phase 3: BotGuard & Anti-Bot Architecture
- **Action**: Created `botguard-provider` Node.js microservice in docker-compose.
- **Reasoning**: To safely execute the `api/jnn/v1/GenerateIT` challenge logic within a sterile JSDOM environment, isolating it from Python to prevent anti-logger traps and thread-blocking.
- **Action**: Installed `Camoufox` browser engine and created `camoufox_session.py`.
- **Reasoning**: To spoof the `WEB` client identity while entirely evading JavaScript prototype checks (e.g., `navigator.webdriver`). Camoufox patches these signatures at the C++ level.
- **Action**: Implemented `oauth_device_flow.py`.
- **Reasoning**: To generate datacenter-safe OAuth tokens using the `TVHTML5` client identity, preventing the accidental leakage or banning of residential cookies in a datacenter environment.

## Phase 4: Upload Automation & Quota Management
- **Action**: Installed `redis` microservice alongside python `credential_pool.py`.
- **Reasoning**: To safely pool and rotate multiple GCP OAuth tokens across multiple stateless worker nodes, gracefully falling back to in-memory environment variables if the database is unreachable.
- **Action**: Enforced `[height<=1080]` format cap in `engine.py` and implemented One-Shot POST via `uploader.py`.
- **Reasoning**: Ensures file sizes remain small enough to safely upload in a single HTTP POST without triggering memory timeouts, circumventing the need for complex Resumable Upload chunking.
- **Action**: Scaffolded `websocket_manager.py`.
- **Reasoning**: Delivers the final `unlisted` YouTube Watch URL directly to the frontend in real-time, completely eliminating the need for synchronous HTTP polling.

## Phase 5: Frontend UI & API Integration
- **Action**: Modified `index.html` and `tailwind.config.js`.
- **Reasoning**: Added the Inter font and pulsing gradient keyframes to fulfill the strict requirement for premium aesthetics and modern dynamic design.
- **Action**: Created `useWebSocket.js` custom React hook.
- **Reasoning**: To establish a lightweight, dependency-free real-time communication channel with the backend, bypassing the need for heavy global state libraries.
- **Action**: Implemented the UI in `App.jsx` using `index.css` glassmorphism classes.
- **Reasoning**: To deliver a "WOW" factor interface with translucent panels and dynamic background gradients, satisfying the high-quality modern UI constraint.
- **Action**: Created `backend/src/main.py`.
- **Reasoning**: Serves as the central FastAPI entry point, exposing the `/api/start-pipeline` REST endpoint to trigger background tasks while independently managing the `/ws/pipeline` WebSocket feed.

## Phase 6: Deployment & IP Rotation Topology
- **Action**: Modified `docker-compose.yml` to set `network_mode: "host"` on the backend container.
- **Reasoning**: To bypass Docker's default IPv4 NAT isolation. This allows the container's network stack to merge with the host, ensuring the Python worker natively inherits the rotating IPv6 egress addresses generated by the host cronjob.
- **Action**: Injected dynamic residential proxy into `engine.py` while explicitly unsetting it for `aria2c`.
- **Reasoning**: Satisfies the strict routing constraint. BotGuard metadata challenges execute over the residential proxy ($15/GB), while the 4GB binary payloads download via the cheap, rotating host IPv6 stack ($0.60/GB).
- **Action**: Scaffolded `rotate_ipv6.sh` and `nginx.conf`.
- **Reasoning**: Automates the SLAAC evasion via `ip -6 addr add` binding every 30 minutes, and secures the Host-networked Docker container behind a robust reverse proxy with WebSocket upgrade support.
